05.其他问题

函数返回局部变量的指针是否安全

  • 是安全的。Go的内存管理机制会自动扩展局部变量的生命周期
  • 底层使用runtime_newObject会把局部变量封装成堆变量

如何从panic中恢复？或者说如何拦截panic？

  • 使用recover函数可以从panic中恢复。通常配合defer使用。
  • recover只能在发生panic时使用，并且只能捕获当前goroutine中的panic。
    func safeFunc() {
  defer func() {
      if r := recover(); r != nil {
          fmt.Println("Recovered from panic:", r)
      }
  }()
  panic("something went wrong")
}

Golang中make和new的区别

  • new：分配内存，但不初始化。它返回的是类型的指针，通常用于值类型（如数组、结构体）。
  • make：只用于创建slice、map和channel，并且会初始化这些数据结构，返回的不是指针而是数据类型的引用。

什么是闭包(closure)？可以在哪些地方使用？

  • 闭包是在函数内声明的匿名函数，可以捕获和引用外部作用域变量。常用于回调函数或工厂函数。
  • 编译的时候就可以确定哪些变量会被闭包函数引用，这个时候使用runtime_newobject生成堆变量
    func counter() func() int {
  count := 0
  return func() int {
      count++
      return count
  }
}

什么是Data Race？go中如何定位和避免？

  • Data Race [竞赛]，内存竞争

    Data Race 是指两个或多个 goroutine 并发访问共享变量，并且至少一个是写操作，且这些操作没有适当的同步机制。 使用 -race 选项检测数据竞争。

避免方法：

  • 使用互斥锁 sync.Mutex 来保护共享资源。
  • 使用 channel 进行消息传递，避免直接共享数据。

Go 语言的 sync.Mutex 由两个字段 state 和 sema 组成

  • state 表示当前互斥锁的状态
  • sema 是用于控制锁状态的信号量。

sync.Mutex 有两种模式 — 正常模式和饥饿模式

  • 正常模式下，锁的等待者会按照先进先出的顺序获取锁
  • 刚被唤起的 Goroutine 与新创建的 Goroutine 竞争时，大概率会获取不到锁
  • 为了减少这种情况的出现，一旦 Goroutine 超过 1ms 没有获取到锁
  • 它就会将当前互斥锁切换饥饿模式，防止部分 Goroutine 被『饿死』
  • 在饥饿模式中，互斥锁会直接交给等待队列最前面的 Goroutine。
  • 新的 Goroutine 在该状态下不能获取锁、也不会进入自旋状态，它们只会在队列的末尾等待。
  • 如果一个 Goroutine 获得了互斥锁并且它在队列的末尾或者它等待的时间少于 1ms，
  • 那么当前的互斥锁就会切换回正常模式。

sync.Mutex 的内部结构

type Mutex struct {
    state int32
    sema  uint32
}
  • state 是锁的状态标志位，包含两部分信息：
    • 最低位：锁是否被占用（locked 位）。
    • 其他位：表示等待者数量（woken, starving 等，组合出完整状态）。
  • sema 是用于协调阻塞和唤醒 goroutine 的信号量。

加锁（Lock）流程

  • 尝试 CAS 抢占锁（乐观锁）
    if atomic.CompareAndSwapInt32(&m.state, 0, mutexLocked) {
  return // 加锁成功，无竞争
}
  • 失败 → 表示锁被占用，进入“慢路径”
    • 把自己标记为等待者。
    • 阻塞当前 goroutine（挂起），等待解锁后被唤醒。
    • 如果处于 "starvation mode"（饥饿模式），那么直接把锁传递给下一个等待者

解锁（Unlock）流程

  • 原子地将 locked 标志位清除（即设置为 0）。
  • 如果还有等待者，则唤醒一个
  • 支持“饥饿模式”：长时间未获得锁的 goroutine 会被优先唤醒，避免“饿死”。

饥饿模式（Starvation Mode）

Go 的 Mutex 为了避免某些 goroutine 一直拿不到锁，引入了饥饿模式策略

  • 如果某个 goroutine 等待超过 1ms，会将锁转入饥饿模式。
  • 在饥饿模式下
    • 当前锁释放时，锁会直接交给最早等待的 goroutine。
    • 不再让新来的 goroutine 竞争，直到没有等待者为止。

底层机制：信号量 + 调度器挂起唤醒

底层是依靠 Go runtime 的信号量原语（runtime_Semacquire, runtime_Semrelease）实现阻塞与唤醒

  • 不会一直自旋浪费 CPU。
  • 实际是通过调度器挂起 goroutine，释放 CPU 使用权，等锁可用时再恢复运行。
Lock() →
 ┌────────────┐
 │ state = 0? │──────► 成功加锁 (返回)
 └────────────┘
       │
       ▼
等待队列中挂起当前 goroutine (sema)
       │
       ▼
   等待 Unlock 唤醒
