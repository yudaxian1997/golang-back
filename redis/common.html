05.其他问题

缓存穿透、缓存击穿、缓存雪崩

  • 缓存穿透（数据库中也不存在的 key）: 缓存空对象（如 null）+ 设置较短过期时间, 使用布隆过滤器拦截无效 key, 增加接口层校验（防止非法请求）
  • 缓存击穿（热点 key 失效瞬间并发访问 DB）: 使用互斥锁（如 Redis 分布式锁）, 设置随机过期时间（避免同一时间失效）, 或使用永不过期 + 后台异步更新策略

  • 缓存雪崩（大量 key 同时过期）: 给缓存设置 过期时间的随机偏移, 构建多级缓存（本地 + Redis）, 限流 + 降级 + 熔断策略兜底

  • 预热: 利用redis回写机制同步到redis。发布了之后手动点一下界面，把数据加载到redis。使用程序添加。

  • 雪崩：key失效之类的，请求全部访问到数据库。永不过期，或者时间错开。
  • 穿透：redis 数据库都没有数据。空对象缓存 + 布隆过滤器
  • 击穿：大量请求同时访问一个key的时候，这个key刚好失效，导致流量访问到数据库，一般是热点key。差异失效时间，双检查更新策略

Redis MoreKey问题，1000w数据量遍历

  • 使用SCAN相关的命令来迭代数据, 不要使用key * 线上环境需要禁用或者重命名这个命名
  • SCAN cursor [Pattern] [Count]

BigKey问题

  • 一般大的是key导致的大value的问题
  • string类型控制再10k以内，hash list set zset 元素个数不要超过5000个
  • 使用 --bigkey 或者 memory usage 来统计bigkey
  • 调优配置: LAZY FREEING 使用 unLink 异步删除, del 命令是阻塞删除

缓存双写一致性的问题

  • 双检加锁： 查询 -> 加锁 -> 再查询，避免多线程都访问数据库
  • 先更新数据库，再更新缓存
  • 先更新缓存，再更新数据库
  • 先删除缓存，再更新数据库 延时双删除
  • 先更新数据库，再删除缓存 [正确逻辑]

Redis分布式锁

独占，高可用，防止死锁，不乱抢，重入性

  • setnx + expire => 如果键不存在就添加记录
  • del的时候判断线程id是否是自己的锁，是自己的才可以删除
  • 查找和删除并不是原子的，需要使用lua脚本来操作
  • 可重入锁设计 添加一个计数器，所以setnx不行，需要使用hset记录重入次数
  • 自动续期功能 添加一个扫描程序
