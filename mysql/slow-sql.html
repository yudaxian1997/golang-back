07.slow sql

MySQL慢查询优化的目标是提升查询效率、减少系统负载

一、定位慢查询

  • 开启慢查询日志 ```sql -- 临时开启 SET GLOBAL slow_query_log = ON; SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log'; SET GLOBAL long_query_time = 1; -- 超过1秒的SQL会被记录

-- 查看当前配置 SHOW VARIABLES LIKE '%slow%';


#### 二、分析查询性能

- 使用 EXPLAIN 查看执行计划，分析是否走了索引、是否存在全表扫描。
```sql
EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';
  • 重点字段说明:
    • type：连接类型（ALL最差，最好是 const、ref）
    • rows：扫描的行数
    • Extra：如出现 "Using filesort"、"Using temporary"，说明可能存在性能问题

三、常见慢查询优化方案

  • 添加或优化索引
    • 建在 WHERE、JOIN、ORDER BY、GROUP BY 涉及的列上
    • 避免在索引列上使用函数（如 WHERE DATE(created_at) = '2024-01-01'）
    • 避免在索引列上进行运算（如 WHERE score + 1 = 100）
    • 对多列使用联合索引时，遵循最左前缀原则
  • 查询语句优化
    • 避免 SELECT *，只选需要的列
    • 拆分复杂的子查询或嵌套查询
    • 使用 EXISTS 替换 IN（尤其当子查询结果较大时）
    • 对分页查询使用主键或索引列作为条件优化大offset场景
  • 表结构优化
    • 选择合适的数据类型（如 INT 替代 VARCHAR）
    • 合理拆分大表：垂直拆分（字段）、水平拆分（行）
    • 使用分区表（适用于大数据量按范围查询）

为什么在 MySQL 中选择 INT 而不是 VARCHAR 来存储类似 ID 的数据？

性能 + 存储空间 + 索引效率。

对比和排序效率更高

  • INT 是定长、直接使用 CPU 指令比较
  • VARCHAR 是变长、逐字符比较、可能涉及字符集编码解码

存储空间更小

类型 字节数（存储）
INT 4 字节
BIGINT 8 字节
VARCHAR(20) 21 字节（额外加1字节长度信息）
  • 假如你用 VARCHAR(20) 存储一个 ID，它实际占用比 INT 多约 5倍 空间。
  • 这会直接影响：
    • 数据表大小
    • 缓存命中率（数据页越小，一页装更多记录）
    • IO 性能（磁盘和内存读写更高效）

索引效率更高

  • 索引底层通常是 B+Tree：
    • 数值类型（INT）更紧凑，树更“矮”，查找路径短
    • 字符串（VARCHAR）变长，对比慢，树更“高”，查找路径长

即使使用了索引，LIMIT offset, count 在 offset 很大的情况下也可能会很慢

核心答案：B+Tree 索引是“有序 + 可定位”的结构，id > 100000 可以直接跳到目标节点开始扫描

WHERE id > 100000 的行为

SELECT * FROM users WHERE id > 100000 ORDER BY id LIMIT 10;
  • 因为 id 是索引字段（主键或建了索引）
  • MySQL 会
    • 直接跳转到值为 100000 的节点（或比它大的第一个）
    • 从那里开始扫描 10 条记录
  • 所以它不需要扫描或处理前面的 100000 条记录！

而 LIMIT 100000, 10 的行为：

SELECT * FROM users ORDER BY id LIMIT 100000, 10;
  • 即使 id 有索引，它仍然必须：
    • 按照索引顺序 逐条扫描 100000 条记录
    • 丢弃它们
    • 然后返回接下来的 10 条
  • 它不能“跳过去”，因为你没有给出跳转位置的条件！
查询方式 是否跳过前面数据 是否快 说明
WHERE id > ? LIMIT N ✅ 是（索引跳转） ✅ 快 可直接定位 + 顺序扫描
LIMIT offset, count ❌ 否（逐条扫描） ❌ 慢 仍然要扫描、丢弃前 offset 条
