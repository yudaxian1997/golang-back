04.xa&&tcc

分布式事务, XA 协议和 TCC 模式

XA 是一种两阶段提交协议（2PC），由 X/Open 组织提出，用于在多个资源之间协调提交/回滚

  • 各参与者（数据库节点）接收到事务指令，执行 SQL 操作，但不提交，仅写入 redo/undo 日志。
  • 返回“可以提交”或“失败”。
  • 如果所有参与者都返回“准备就绪”，协调者发送 commit 指令，各节点执行提交。
  • 如果有一个失败，则协调者发出 rollback，全部回滚。
    优点
  • 通用：被各大数据库广泛支持（MySQL InnoDB、Oracle 等）。
  • 强一致性：要么全部成功，要么全部失败。
    缺点
问题 描述
阻塞 如果协调者宕机，参与者可能会长时间锁定资源（等待 commit/rollback）
性能开销大 每阶段都涉及日志写入、锁资源，适合金融等高一致性场景
依赖数据库支持 要求资源参与者支持 XA 接口，如 InnoDB 引擎中的 XA START, XA END

TCC 模式详解（Try-Confirm-Cancel）

  • TCC 是一种业务层的事务解决方案，通过三个明确步骤协调资源：
  • Try：尝试性操作，做预留资源（如冻结余额、不做实操作）
  • Confirm：正式提交，执行实际动作（扣款/扣库存等）
  • Cancel：取消事务，释放资源或回滚 Try 中的操作
优点 描述
非阻塞 没有锁，支持高并发、高吞吐
业务可控 Try/Confirm/Cancel 都由开发者定义，灵活性高
更易于故障恢复 各阶段可幂等处理，重试机制更完善
缺点 描述
实现复杂 每个参与服务都需开发 Try/Confirm/Cancel 三个接口
幂等性 & 空回滚处理 每个阶段需支持幂等 & 补偿逻辑
对业务入侵较强 会改变原有服务结构，开发成本大
框架 支持模型 核心特点 适用场景
DTM ✅ TCC / Saga / XA / 事务消息 开源、文档全，支持 HTTP/gRPC；事务模式最全 ✅ 推荐使用，成熟稳定
Seata (Go 支持) TCC / AT / Saga 原阿里开源，Go 支持不完善（实验性），重度依赖 Java 原版 不推荐，生态不成熟
go-saga Saga 极简轻量、无第三方依赖，主要用于学习或定制开发 简单链式事务，控制粒度较粗
gocommit TCC（早期实验） 简单的 TCC 框架，已较久未维护 学习用
Watermill 消息驱动（Kafka/NATS） 类似 Spring Cloud Stream，在消息队列中实现事务保障（幂等 + 重试） 事件驱动系统，适合异步场景
