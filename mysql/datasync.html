06.data sync

优化大批量数据写入性能

数据库层优化策略:

  • 使用批量写入（Batch Insert）
  • 禁用或延迟二级索引
    • 每次写入时都会更新所有索引。可在同步前临时去除非必要的二级索引，同步完成后再重建
  • 控制事务大小
  • 关闭或优化日志写入

应用侧优化方案

  • 并发写入（分片并发）
  • 异步写入 + 队列缓冲
    • 使用 Kafka/RabbitMQ/NATS 等做 buffer，异步写入数据库
    • 提高写入平滑性，避免短时间高并发压垮数据库

高效的数据一致性校验机制

场景 描述
主从数据库一致性 MySQL 主从、副本集是否同步成功
数据同步工具一致性 Kafka → Elasticsearch，Canal → ClickHouse 等
数据迁移后验证 如 Redis → TiKV，MySQL → OceanBase
缓存与源数据库一致性 Redis vs MySQL 等
跨系统校验 OLTP vs OLAP 比如 Hive vs MySQL

校验策略

  • 全量校验：使用 checksum 或 count 比对
  • 增量校验：根据时间戳或主键范围分片校验
  • 实时校验：通过 binlog 位点确保同步准确性

方案 1: 基于主键范围 + CRC/Hash 校验

  • 将数据按照主键范围切分，如 ID 010W，10W20W...
  • 分别计算 MySQL 和目标库的哈希（如 CRC32、MD5、XXHash）
  • 哈希一致 → 校验通过；不一致 → 再深入行级校验

方案 2: 基于聚合校验字段

  • 比较行数、字段聚合值是否一致
  • 快速判断“是否大致一致”

方案 3: 基于版本字段 + 增量校验

  • 适用于有 updated_at 字段的系统
  • 每次只校验过去 N 分钟/小时的更新记录
  • 记录 last_check_time，不断推进

在大规模数据同步

  • 数据传输效率、批量操作、并发控制、容错设计、系统架构等多方面进行优化
  • 优化同步性能的基本原则

    减少每次操作的开销

  • 批量写入：减少每次写入的次数，每批写入尽可能多的记录，减少与数据库的交互次数。
  • 异步写入：同步过程中的数据处理和网络传输等步骤尽量异步化，避免阻塞。

    提高并发度

  • 利用并行处理和分片技术，通过多线程或多进程提升吞吐量。

    减少数据传输量

  • 使用高效的数据压缩算法（如 zlib、Snappy、LZ4）减少传输数据的体积。
  • 采用增量同步方式，仅同步新增或更新的数据。

MySQL 主从复制的原理，以及可能遇到的问题和解决方案

  • 主库的 Binlog（Binary Log）主库记录所有的数据修改操作（如 INSERT、UPDATE、DELETE）到其 二进制日志（binlog） 中，所有更改都会依赖此日志。
  • 从库的 IO 线程 IO 线程连接到主库时，会向主库请求 binlog 文件 和 位点（log position），然后从主库读取新的 binlog 数据。
  • 从库的 SQL 线程 从库的 SQL 线程 会读取 IO 线程存储的 binlog 数据，并执行相应的 SQL 语句来更新从库的数据库内容。
  • 复制的异步性质 主库的数据更新可能会先于从库的同步。这导致了主从之间可能会存在数据不一致的情况（如延迟、丢失数据）。
  • 复制的延迟 在高并发的环境下，从库可能会落后于主库，因为它需要顺序执行从主库接收到的 binlog 事件。

常见问题与解决方案

复制延迟
  • 主库的写入操作过于频繁，导致从库跟不上。
  • 网络带宽不足，导致 binlog 传输的速度较慢。
  • 从库的硬件性能较差，处理速度较慢。
  • 从库的 SQL 线程处理速度慢（比如执行复杂查询时）。
  • 增加从库数量：如果延迟较大，可以增加从库数量，分担负载。
  • 优化主库的写入性能：优化主库的 SQL 语句，减少锁竞争，避免长时间的锁。
  • 硬件升级：提升从库的 CPU、内存、IO 性能，优化从库的磁盘 IO。
  • 优化网络带宽：确保主从服务器之间的网络带宽足够，避免网络拥堵。
  • 调整复制参数：如调整 sync_binlog、innodb_flush_log_at_trx_commit 等参数来提升写性能。
复制丢失数据（数据不一致）
  • 配置持久化 binlog 文件：确保主库的 binlog 文件在主库重启时不会丢失，修改 expire_logs_days 参数来配置日志的保留时间。
  • 使用 GTID（Global Transaction Identifier）：GTID 是一种全局唯一标识事务的标识符，开启 GTID 模式后可以避免一些数据丢失问题。
  • 通过使用 GTID，MySQL 可以确保每个事务在复制过程中都有一个明确的标识，避免了传统基于 binlog 位置的复制方式可能带来的数据错位和丢失。
