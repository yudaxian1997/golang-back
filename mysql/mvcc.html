01.mvcc

MVCC (多版本并发控制)

  • MVCC 是数据库常用的一种并发控制机制，它通过保存数据的多个版本，在读写之间尽可能避免加锁，从而提高并发性能。
  • MVCC 通过给每一行记录维护两个隐藏的版本号字段
字段 说明
trx_id（事务ID） 最后一次修改该行数据的事务 ID（插入或更新）
roll_pointer（回滚指针） 指向 Undo Log 的指针，可以查到旧版本数据

另外还有 当前事务的两个 ID 值：

  • 当前事务 ID：由事务开始时分配
  • ReadView：当前事务可见的其他事务的 ID 快照（MVCC 的核心视图）

InnoDB 中 MVCC 的实现机制

    1. 两个隐藏字段: 在 InnoDB 的每一行数据中，都会有两个隐藏字段 trx_id roll_pointer
    1. Undo Log（回滚日志）: 每次更新、删除数据时，都会生成一条 undo log 来保存修改前的版本
    1. ReadView（快照视图）:
    2. 在事务启动时生成一个 ReadView
    3. 包含当前系统中活跃事务的 ID 集合（未提交事务）
    4. 用于判断某行记录的 trx_id 是否对当前事务可见

MVCC 和事务隔离级别的关系（MySQL InnoDB）

隔离级别 是否使用 MVCC 说明
Read Uncommitted ❌ 否，用原始数据，无快照 能读到未提交数据，不用 MVCC
Read Committed ✅ 每次读取新生成 ReadView 快照读每次都更新，不能读旧事务未提交数据
Repeatable Read ✅ 事务开始生成 ReadView，不变 整个事务期间快照一致，可重复读（默认）
Serializable ❌ 强制加锁，串行化执行 不使用 MVCC，直接加锁实现一致性

示例

  • 01.假设数据库中有一条记录：id: 1, name: 'Alice', trx_id: 80
  • 02.而当前事务 ID 是 100，事务刚开始，系统中存在如下活跃事务：90, 95, 99 （正在运行但未提交）
    ReadView = {
 min_trx_id = 90
 max_trx_id = 99
 active_trx_ids = {90, 95, 99}
}
  • 0.3 SELECT * FROM users WHERE id = 1;
    • Step 1：读取主键索引，找到主记录（主键 id=1）name = 'Alice', trx_id = 80
    • Step 2：判断该版本是否对当前事务可见？trx_id = 80 ReadView.min_trx_id = 90 80 < 90 => 可见
      • 这个数据版本是在当前事务开始之前已经提交的事务写入的，对当前事务是可见的 → 返回该数据。
      • 如果数据是 trx_id = 95（未提交的事务写的）怎么办？
        • 95 ∈ ReadView.active_trx_ids => 不可见
        • 因为写入数据的事务还未提交，对当前事务不可见
        • 则根据 roll_pointer 找到 undo log，获取上一个版本的数据，再次判断。
