10.安全

重入攻击

攻击者通过 fallback 函数多次调用合约函数，绕过更新逻辑，重复取款。

防御策略

  • 遵循 Checks-Effects-Interactions 模式 检查-影响-交互模式
  • 使用 reentrancyGuard 修饰器（如 OpenZeppelin） 重入锁
  • 更新状态后再转账

选择器碰撞

由于函数选择器只有4字节，非常短，很容易被碰撞出来：即我们很容易找到两个不同的函数，但是他们有着相同的函数选择器

签名重放

  • 普通重放：将本该使用一次的签名多次使用
  • 跨链重放：将本该在一条链上使用的签名，在另一条链上重复使用
  • 将使用过的签名记录下来，比如记录下已经铸造代币的地址 mintedAddress，防止签名反复使用
  • 将 nonce （数值随每次交易递增）和 chainid （链ID）包含在签名消息中，这样可以防止普通重放和跨链重放攻击

坏随机数

我们通常使用预言机项目提供的链下随机数来预防这类漏洞，例如 Chainlink VRF。这类随机数从链下生成，然后上传到链上，从而保证随机数不可预测

绕过合约长度检查

很多 freemint 的项目为了限制科学家（程序员）会用到 isContract() 方法，希望将调用者 msg.sender 限制为外部账户（EOA），而非合约

    // 利用 extcodesize 检查是否为合约
    function isContract(address account) public view returns (bool) {
        // extcodesize > 0 的地址一定是合约地址
        // 但是合约在构造函数时候 extcodesize 为0
        uint size;
        assembly {
            size := extcodesize(account)
        }
        return size > 0;
    }

这里有一个漏洞，就是在合约在被创建的时候，runtime bytecode 还没有被存储到地址上，因此 bytecode 长度为0。也就是说，如果我们将逻辑写在合约的构造函数 constructor 中的话，就可以绕过 isContract() 检查。

function realContract(address account) public view returns (bool) {
        return (tx.origin == msg.sender);
    }
  • 如果调用者为 EOA，那么tx.origin和msg.sender相等；如果它们俩不相等，调用者为合约
    • tx.origin 最初发起交易的外部账户地址（始终是一个 EOA，永远不会是合约）
    • msg.sender 当前函数调用者，可以是合约也可以是用户账户（EOA）
  • 不要用 tx.origin 做身份验证
    • 这种写法存在 严重安全漏洞，叫做 tx.origin 钓鱼攻击（phishing attack）。
